이 문서는 UX/설계 관련 의견을 냈다가 Claude와 함께 한번쯤 검토해본 내용들의 기록이다. 코드가 바뀌었는지 여부와 무관하게, "질문 → 논의 → 결론" 흐름이 있었던 사안을 남긴다. 나중에 비슷한 질문을 다시 하게 되거나 관련 결정을 내려야 할 때 과거 판단 근거를 참고하기 위한 용도다.

## 2026-07-14 — 로그인 후 뒤로가기: 로그인 페이지로 돌아가야 하는가, 건너뛰어야 하는가

**질문:** 로그인 완료 후 브라우저 뒤로가기를 눌렀을 때 다시 로그인 페이지가 보이는 게 나은지, 건너뛰고 이전 페이지로 바로 가는 게 나은지.

**논의:** 로그인은 완료되면 되돌아가서 다시 볼 이유가 없는 일회성 액션이라, 뒤로가기 시 로그인 폼이 다시 뜨는 건 "로그인이 풀렸나?" 하는 혼란만 줄 뿐이라고 판단. 일반적인 로그인/OAuth 플로우 관례도 replace를 씀.

**결론:** `buyerLoginAction`/`buyerSignupAction`에서 `redirect(next, RedirectType.replace)` 사용하도록 수정 ([buyer-actions.ts](../src/lib/auth/buyer-actions.ts)). 단, 구글 로그인(`/api/auth/google/callback`)은 `NextResponse.redirect`를 쓰는 HTTP 레벨 리다이렉트라 같은 방식으로 못 고치고, `/login` 진입 자체를 바꿔야 하는 별도 작업이라 이번엔 보류.

## 2026-07-14 — 임시 닉네임/계정 유형 표시가 중복돼도 되는가

**질문:** 헤더에 표시되는 임시 닉네임(이메일 앞부분)이 사용자 간 겹칠 수 있는데, 매장에서 같은 이름으로 주문이 들어오면 혼선이 생기지 않을지.

**논의:** 계정 식별은 이미 `userId`(UUID)로 처리되고 있어 닉네임 중복은 시스템상 문제가 안 됨. 매장에서 헷갈릴 수 있다는 우려는 사실 헤더 닉네임이 아니라 체크아웃 때 입력하는 `Order.customerName` 필드(주문마다 별도 입력)에 해당하는 얘기이고, 실제로는 주문번호로 구분하는 게 맞는 방향.

**결론:** 헤더 닉네임에 중복 방지 로직 불필요. 별도 조치 없음.

**결론:** 사용성 저해 없음으로 판단, 수정하지 않음.

**결론:** 로컬/서버 장바구니가 둘 다 비어있지 않을 때만 "다른 장바구니 발견" 확인창을 띄우도록 [cart-context.tsx](../src/lib/cart/cart-context.tsx) 수정. 구현 중 부가로, 컴포넌트 리마운트(새로고침/서버 재시작 등) 시 이미 로그인된 상태인데도 매번 이 확인창이 재발하는 버그를 발견해 `wasOwned()` 플래그로 같이 수정함.

## 2026-07-14 — 플로우 페이지에 좌측 상단 "뒤로가기" 링크를 통일된 규칙으로 추가할 것인가

**질문:** 체크아웃에서 이전 단계(장바구니)로 돌아가는 방법이 헤더의 장바구니 아이콘뿐인데, 앞으로 이런 플로우 전반에 좌측 상단 뒤로가기 링크를 규칙으로 추가하는 게 어떨지.

**논의:** 이미 `/menu` 페이지에 "← 매장 목록"(`/stores`로) 패턴이 존재해, 이 규칙을 확장하는 것뿐이라 자연스럽다고 판단. 비용도 낮음(아이콘+링크 하나).

**결론:** `/checkout` → `/cart`, `/cart` → `/menu?storeId=...`에 동일 스타일(ArrowLeft 아이콘 + 텍스트)로 추가. 여러 경로로 진입 가능해 "이전 단계"가 애매한 진입점 페이지에는 적용하지 않기로 함. 앞으로도 이 규칙 유지하기로 하고 Claude의 로컬 메모리(`feedback_back_link_convention.md`)에도 별도 기록.

## 2026-07-14 — 뒤로가기 링크가 `location.back()`/`replace()`가 아니라 `push`인 것이 맞는가

**질문:** 새로 추가한 뒤로가기 링크가 실제 브라우저 히스토리를 되짚는 `back()`이나 현재 항목을 대체하는 `replace()`가 아니라 그냥 `push`(새 히스토리 항목 추가)인지 확인.

**논의:** 확인 결과 `<Link href="...">` 기본 동작이라 `push`가 맞음. `back()`은 실제 방문 이력을 추적하는 용도일 때나 맞는 선택이고, 이 링크는 "논리적으로 이전 단계"인 고정 경로로 가는 것뿐이라 실제 이력 추적이 아님 — 그렇다면 이동 경로를 남기는(push) 쪽이 맞다고 사용자가 판단.

**결론:** `push` 그대로 유지, `replace`나 `router.back()`으로 바꾸지 않음.

## 2026-07-14 — /cart의 뒤로가기 링크와 "메뉴 추가하기" 버튼이 같은 목적지라 중복 아닌가

**질문:** /cart에 이미 같은 매장 메뉴로 가는 "메뉴 추가하기" 버튼이 있는데, 좌측 상단 뒤로가기 링크까지 유지하는 게 맞는지, 아니면 중복이니 없애는 게 나은지.

**논의:** 규칙의 핵심 가치는 "어느 페이지든 좌측 상단에 뒤로가기가 예측 가능하게 있다"는 일관성인데, 이 페이지만 예외로 빼면 그 일관성이 깨짐. 또한 목적지는 같아도 역할이 다름 — "메뉴 추가하기"는 장바구니 요약 옆의 "더 담기" 액션 버튼, 좌측 상단 링크는 내비게이션 성격. 중복 비용도 낮음(텍스트 링크 하나 추가일 뿐).

**결론:** 둘 다 유지. 같은 목적지로 가는 다른 버튼이 페이지에 있다는 이유만으로 뒤로가기 링크를 빼지 않기로 함.

## 2026-07-24 — 판매자/관리자 로그인 페이지를 역할별로 분리할 것인가

**질문:** 이미 로그인된 판매자/관리자가 홈에서 "판매자·관리자 로그인" 링크(`/admin/login`)를 다시 누르면 세션이 살아있는데도 로그인 폼이 그대로 떠서 로그아웃된 것처럼 보이는 버그를 고치던 중, 근본 원인이 "판매자와 관리자가 로그인 로직/화면을 공유해서"라는 의견이 나옴. 진입 경로 자체를 역할별로 나누는 게 맞지 않은가.

**논의:** 실제로 재현해보니 세션은 멀쩡히 유지되고 있었고, 문제는 `/admin/login`이 인증 게이트 예외로 빠져있어 이미 로그인된 사용자도 검사 없이 로그인 폼을 그대로 보여준다는 점이었음 — 역할이 하나로 합쳐져 있어서 생긴 문제가 아니라, 단순히 "이미 로그인돼 있으면 대시보드로 리다이렉트"하는 처리가 빠진 것. 판매자/관리자가 화면(주문 목록, 헤더 등)을 실제로 공유하고 있는 부분까지 억지로 나누면 유지보수 비용만 늘어나므로, 이번 버그는 로그인 페이지 자체에 세션 체크만 추가해 해결하기로 함.

**결론:** [admin/login/page.tsx](../src/app/(admin)/admin/login/page.tsx)에서 세션을 확인해 이미 seller/platform_admin이면 각자의 기본 화면(`/admin/orders` / `/admin/stores`)으로 리다이렉트하도록 수정. 판매자/관리자 진입 경로 전체를 물리적으로 분리하는 더 큰 구조 변경은 이번 스코프에서 제외하고 별도 논의로 미룸.

## 2026-07-24 — 판매자 주문 목록에서 상태 변경을 어떤 방식의 버튼으로 처리할 것인가

**질문:** 판매자가 주문 상세 페이지에 매번 들어가지 않고도 목록에서 바로 "주문 수락 → 조리완료 → 배달시작 → 배달완료"를 처리할 수 있게 하고 싶은데, 목록 행마다 어떤 형태의 버튼을 두는 게 좋은지. 배달의민족/쿠팡이츠 사장님 앱은 이걸 어떻게 하는지도 참고.

**논의:** 상세 페이지의 `OrderStatusButtons`처럼 5단계 상태를 전부 버튼으로 늘어놓는 방식은 목록처럼 빠르게 훑으며 처리하는 용도에는 과함. 배달앱들은 보통 카드마다 큰 버튼 하나를 두고 현재 상태에 따라 라벨이 바뀌는 방식(예: "주문 수락"→"조리 완료"→"배달 시작"...)을 쓰고, 최초 접수 단계에서만 수락/거절 갈림길을 추가로 둔다는 게 일반적으로 알려진 패턴. 이 프로젝트의 `OrderStatus`에는 애초에 거절/취소 개념이 없어(선형 5단계뿐), 거절 플로우를 넣으려면 상태값 자체를 확장해야 하는 별도 스코프였는데, 사용자가 이번에 바로 추가하기로 결정.

**결론:** `OrderStatus`에 `cancelled`("거절됨") 추가. 목록 행([order-list.tsx](../src/components/order-list.tsx))에는 상태별 다음 액션 라벨을 보여주는 단일 버튼(`ORDER_NEXT_ACTION`, [order-row-actions.tsx](../src/components/order-row-actions.tsx))만 두고, "접수됨" 상태일 때만 옆에 "거절" 버튼을 추가로 노출 — 조리를 이미 시작한 주문은 취소가 아니라 완료까지 진행하는 게 맞다고 판단해 거절은 접수 단계에서만 허용. 상세 페이지의 `OrderStatusButtons`에도 동일 조건으로 거절 버튼을 추가해 두 화면의 규칙을 일치시킴. `OrderStatusTimeline`은 `cancelled`를 선형 진행바의 한 단계가 아니라 별도 배너로 표시하도록 분기 처리(구매자 확인 페이지에도 동일하게 반영됨).
