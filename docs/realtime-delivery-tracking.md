# 실시간 배달 내역 추적 — 구현 계획

각 주문의 상태 변화를 시간순 내역(타임라인)으로 남기고, 구매자·판매자가 실시간으로 볼 수 있게 만드는 기능의 작업 계획. 이 기능을 여러 세션에 걸쳐 진행할 때 매번 처음부터 재논의하지 않도록 여기에 우선순위와 단계를 고정해둔다.

작업을 시작/재개할 때는 이 문서를 먼저 읽고, 진행 상황에 따라 체크박스를 갱신한다. 완료된 단계는 지우지 말고 `[x]`로 남겨서 이력을 유지한다.

관련 문서: [백로그 — 실시간 주문 상태 추적(SSE 인프라 가이드)](backlog.md). 이 문서는 그 인프라 위에서 실제로 만들 기능의 구체적인 실행 계획이다. SSE/이벤트버스 아키텍처 자체(단일 프로세스 EventEmitter, 배포 체크리스트 등)는 backlog.md를 따르고 여기서 중복 서술하지 않는다.

---

## 전제 확인 (현재 상태, 2026-07-22 기준)

- `Order.status`는 현재 값만 저장하는 단일 필드. **상태가 언제 바뀌었는지 이력이 전혀 없음.** "배달 내역"을 보여주려면 이력 테이블이 먼저 필요함 — 여기가 출발점.
- 판매자는 [order-status-buttons.tsx](../src/components/order-status-buttons.tsx)로 상태를 바꿀 수 있지만, 구매자는 상태를 볼 화면이 전혀 없음.
- 실시간 푸시 인프라(SSE)는 설계만 돼 있고 아직 구현 전.

### 확정된 상태 흐름 (5단계)

기존 4개 값의 이름은 그대로 두고, `ready`(준비완료)와 `completed`(완료) 사이에 `delivering`(배달중) 한 단계를 추가한다. enum에 값을 하나 끼워넣는 것뿐이라 기존 주문 데이터는 손댈 필요 없음(백필 불필요) — 이 변경 이후 새로 만들어지는 주문부터 5단계를 거친다.

```
접수됨(received) → 조리중(preparing) → 준비완료(ready) → 배달중(delivering) [신규] → 완료(completed)
```

## 우선순위 순서 (P0부터 순서대로 진행)

### P0 — 데이터 모델: 5단계 상태 + 변경 이력 저장
- [ ] `OrderStatus` enum에 `delivering` 추가 (`ready`와 `completed` 사이).
- [ ] `OrderStatusHistory` 모델 추가 (Prisma): `id`, `orderId`, `status`, `changedAt`. `Order`와 1:N.
- [ ] 마이그레이션 생성 및 적용.
- [ ] `ORDER_STATUS_LABELS`, `order-status-buttons.tsx`의 상태 목록에 "배달중" 반영.
- [ ] `createOrder` 액션에서 주문 생성 시 최초 이력 행(`status: "received"`) 함께 기록.
- [ ] `updateOrderStatus` 액션에서 상태 변경마다 이력 행 추가.
- [ ] `getOrderStatusHistory(orderId)` 쿼리 함수 작성 (시간순 정렬).
- [ ] 임시로 `/admin/orders/[orderId]`에 이력을 그대로 나열하는 단순 목록 추가 (P3에서 정식 타임라인으로 교체 예정 — 지금은 P0이 실제로 기록되는지 눈으로 확인하기 위한 용도).

**이 단계 없이는 "내역"이라는 개념 자체가 성립하지 않으므로 반드시 먼저 완료.**

**테스트 방법:** 관리자로 로그인 → 아무 주문 상세 페이지에서 상태 버튼을 "조리중"→"준비완료"→"배달중"→"완료" 순서로 눌러본다. 누를 때마다 페이지 하단 임시 이력 목록에 줄이 하나씩 쌓이면 통과.

### P1 — 실시간 인프라 (backlog.md 가이드 적용)
- [ ] `src/lib/events.ts`: 서버 내 이벤트버스(EventEmitter 싱글턴), `emitOrderEvent(orderId, event)` 헬퍼.
- [ ] `updateOrderStatus`에서 이력 기록 후 이벤트 발행.
- [ ] SSE 라우트 핸들러 `/api/orders/[orderId]/stream` — 세션이 해당 주문의 구매자(또는 담당 매장의 판매자/관리자)인지 확인 후 구독 허용.
- [ ] 클라이언트 공용 훅(`useOrderStream` 등): `EventSource` 구독, 메시지 수신 시 상태/이력 갱신, 재연결 시 스냅샷 재조회로 유실 구간 보정.

**테스트 방법:** 브라우저 개발자도구 Network 탭에서 `/api/orders/{id}/stream` 요청이 `eventsource` 타입으로 계속 연결된 상태(pending)인지 확인. 다른 탭(또는 관리자 화면)에서 그 주문 상태를 바꾸고, 새로고침 없이 이 탭에 이벤트가 도착하는지(Network 탭의 EventStream 서브탭에 메시지가 찍히는지) 확인.

### P2 — 구매자용 배달 추적 UI
- [ ] `/confirmation/[orderId]`에 상태 타임라인 컴포넌트 추가 — 지난 이력은 타임스탬프와 함께, 현재 상태는 강조 표시.
- [ ] P1의 훅을 연결해 새로고침 없이 실시간 갱신.
- [ ] (선택, 범위 확장 시) "내 주문" 목록 페이지 신설 — 진행 중인 주문을 모아보기. 별도 항목으로 분리 가능.

**테스트 방법:** 구매자 계정으로 주문 후 확인 페이지를 열어둔 채로, 다른 브라우저(또는 시크릿창)에서 판매자로 로그인해 그 주문 상태를 단계별로 바꿔본다. 구매자 화면이 새로고침 없이 실시간으로 단계가 넘어가면 통과.

### P3 — 판매자/관리자 반영
- [ ] 매장별 실시간 신규 주문 피드(backlog.md의 기존 항목)와 통합 — 같은 이벤트버스를 재사용.
- [ ] `/admin/orders/[orderId]` 주문 상세 페이지의 P0 임시 이력 목록을 정식 타임라인 UI로 교체.

**테스트 방법:** 새 주문을 넣었을 때 판매자 화면에 새로고침 없이 목록 맨 위로 올라오는지, 주문 상세의 이력 타임라인이 P0 임시 목록과 같은 데이터를 더 보기 좋은 형태로 보여주는지 확인.

### P4 — 배포 전 점검
- [ ] backlog.md의 원격 배포 체크리스트(단일 프로세스 구동, 리버스 프록시 버퍼링 해제 등) 재확인.

**테스트 방법:** 실제 배포 환경(또는 그에 준하는 로컬 재현)에서 P1~P3 테스트를 다시 한번 반복.

## 미정 사항 (착수 전 결정 필요)

- ~~상태 세분화 여부~~ → **결정됨**: `delivering`(배달중) 추가, 5단계로 확정 (위 "확정된 상태 흐름" 참고).
- **위치 기반 추적 포함 여부:** "배달 내역"이 상태 텍스트 타임라인만 의미하는지, 아니면 실제 배달원 위치(지도)까지 포함하는지. 후자는 완전히 별도 규모의 작업(배달원 앱/위치 전송 주체가 없음)이라 이번 범위에서는 제외하는 게 합리적으로 보이나 확인 필요.
- **P2의 "내 주문" 목록 페이지**를 이번 스코프에 포함할지, 확인 페이지 실시간 배지만으로 먼저 끝낼지.
