# 실시간 배달 내역 추적 — 구현 계획

각 주문의 상태 변화를 시간순 내역(타임라인)으로 남기고, 구매자·판매자가 실시간으로 볼 수 있게 만드는 기능의 작업 계획. 이 기능을 여러 세션에 걸쳐 진행할 때 매번 처음부터 재논의하지 않도록 여기에 우선순위와 단계를 고정해둔다.

작업을 시작/재개할 때는 이 문서를 먼저 읽고, 진행 상황에 따라 체크박스를 갱신한다. 완료된 단계는 지우지 말고 `[x]`로 남겨서 이력을 유지한다.

관련 문서: [백로그 — 실시간 주문 상태 추적(SSE 인프라 가이드)](backlog.md). 이 문서는 그 인프라 위에서 실제로 만들 기능의 구체적인 실행 계획이다. SSE/이벤트버스 아키텍처 자체(단일 프로세스 EventEmitter, 배포 체크리스트 등)는 backlog.md를 따르고 여기서 중복 서술하지 않는다.

---

## 전제 확인 (현재 상태, 2026-07-22 기준)

- `Order.status`는 현재 값만 저장하는 단일 필드. **상태가 언제 바뀌었는지 이력이 전혀 없음.** "배달 내역"을 보여주려면 이력 테이블이 먼저 필요함 — 여기가 출발점.
- 판매자는 [order-status-buttons.tsx](../src/components/order-status-buttons.tsx)로 상태를 바꿀 수 있지만, 구매자는 상태를 볼 화면이 전혀 없음.
- 실시간 푸시 인프라(SSE)는 설계만 돼 있고 아직 구현 전.

### 확정된 상태 흐름 (5단계)

기존 4개 값의 이름은 그대로 두고, `ready`(준비완료)와 `completed`(완료) 사이에 `delivering`(배달중) 한 단계를 추가한다. enum에 값을 하나 끼워넣는 것뿐이라 기존 주문 데이터는 손댈 필요 없음(백필 불필요) — 이 변경 이후 새로 만들어지는 주문부터 5단계를 거친다.

```
접수됨(received) → 조리중(preparing) → 준비완료(ready) → 배달중(delivering) [신규] → 완료(completed)
```

**이 단계 없이는 "내역"이라는 개념 자체가 성립하지 않으므로 반드시 먼저 완료.**

**테스트 방법:** 관리자로 로그인 → 아무 주문 상세 페이지에서 상태 버튼을 "조리중"→"준비완료"→"배달중"→"완료" 순서로 눌러본다. 누를 때마다 페이지 하단 임시 이력 목록에 줄이 하나씩 쌓이면 통과.
→ **확인됨**: 기존 주문에서 4단계 클릭 시 시간과 함께 순서대로 쌓임, 신규 체크아웃 주문에도 "접수됨" 초기 이력이 자동으로 남음.

- [x] `src/lib/events.ts`: 서버 내 이벤트버스(EventEmitter 싱글턴), `emitOrderStatusEvent`/`subscribeToOrder` 헬퍼.

- [x] 클라이언트 컴포넌트([order-status-history.tsx](../src/components/order-status-history.tsx)): `EventSource` 구독, 메시지 수신 시 이력에 새 항목 추가. (재연결 시 스냅샷 재조회 보정은 P2/P3에서 실제 UI 붙일 때 함께 처리 — 지금은 P0 임시 목록에 실시간만 얹은 상태)

**테스트 방법:** 브라우저 개발자도구 Network 탭에서 `/api/orders/{id}/stream` 요청이 `eventsource` 타입으로 계속 연결된 상태(pending)인지 확인. 다른 탭(또는 관리자 화면)에서 그 주문 상태를 바꾸고, 새로고침 없이 이 탭에 이벤트가 도착하는지(Network 탭의 EventStream 서브탭에 메시지가 찍히는지) 확인.
→ **확인됨**: 같은 주문 상세 페이지를 두 탭에 띄우고 한쪽에서 상태를 바꾸니 반대쪽 탭에 새로고침 없이 이력이 즉시 추가됨.
